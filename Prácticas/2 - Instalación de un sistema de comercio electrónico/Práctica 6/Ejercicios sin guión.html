Ejercicios sin guión



Ejercicio 1

La prueba de carga Jmeter test_esfuerzo_tomatocart.jmx, que hemos salvado en el punto 41 del Ejercicio 4, es adecuada para estresar al servidor Tomatocart y ver su comportamiento ante una situación extrema. Sin embargo, este test no es representativo del patrón de accesos generados por seres humanos, porque los accesos entre páginas se suceden de forma instantánea, mientras que una persona necesita al menos unos segundos para identificar el contenido de la página, y un tiempo aún mayor si llega a leer (total o parcialmente) su contenido.

En este Ejercicio se requiere del alumno que modifique el test test_esfuerzo_tomatocart.jmx para que simule la situación generada por un gran número de consultas al catálogo, pero procedente de seres humanos. Para ello se deberán introducir los siguientes cambios:

  • Utilizar Timers aleatorios de Jmeter (Uniform o Gaussian) para generar retardos aleatorios tras el acceso a cada página. Usar valores de entre 1 y 30 segundos, y configurar los temporizadores para que haya al menos una página de retardos breves, y al menos una página de retardos largos.

  • Para compensar la menor frecuencia de los accesos, aumentar el número de usuarios a 50.

Salvar el nuevo test con el nombre test_esfuerzo_humano_tomatocart.jmx. Ejecutarlo y comprobar su efecto sobre el servidor con las mismas herramientas (top, vmstat, y los diversos Listeners Jmeter) vistos en el Ejercicio 4.



Ejercicio 2

Limpiar totalmente el sistema Tomatocart (llevando al servidor debiantomato al estado de la instantánea Tomatocart final, y modificar el test Jmeter bucle_creacion_cuentas.jmx (creado en el punto 31 del Ejercicio 5) para que genere, en ese sistema Tomatocart limpio, un total de 50 cuentas de usuario. Cada cuenta pertenecerá al usuario Prueban, con dirección de e-mail prueban@deii.org, y password deii15prueban, con n tomando valores entre 1 y 50.



Ejercicio 3

Aprovechando las cuentas de usuario creadas, y empleando los mecanismos del proxy Jmeter descritos en los Ejercicios 4 y 5, preparar un test Jmeter que reproduzca la siguiente secuencia:
  • Login de un usuario en el sistema Tomatocart.

  • Navegación por al menos 3 páginas, con retardos aleatorios.

  • Logout
Tras comprobar que el test funcione correctamente para un sólo usuario, automatizarlo para convertirlo en un test capaz de reproducir los accesos de múltiples usiarios simultáneos. Usar variables de usuario, variables aleatorias, y los diversos controladores lógicos de Jmeter para controlar el número máximo de usuarios que puedan hacer login simultáneamente, y para conseguir que las operaciones de login usen el total de las 50 cuentas de usuario creadas en el Ejercicio 2.

Una vez obtenido el test Jmeter deseado, utilizarlo para comprobar la carga del servidor si se permite el acceso a 10, 30 o 50 usuarios simultáneos.



Ejercicio 4

Finalmente, vamos a comparar el comportamiento de Tomatocart en un sólo servidor con la configuración en dos servidores (debiantomato  + debiansql). Para ello, deberemos:
  • Llevar el servidor debiantomato al estado de la instantánea Separado en dos servidores

  • Abrir el servidor debiansql y, sin arrancarlo, crearle una instantánea llamada Separado en dos servidores.

  • Arrancar ambos servidores.

  • Usar los tests Jmeter elaborados en los Ejercicios sin guión 1 a 3, y ejecutarlos contra el sistema Tomatocart en la configuración de dos servidores.

  • Monitorizar (con top, vmstat y los Listeners Jmeter) el rendimiento de ambos servidores.

  • Comparar los resultados obtenidos con los de la configuración de un sólo servidor, y sacar conclusiones acerca de cual de las dos configuraciones ha ofrecido un mejor rendimiento.
